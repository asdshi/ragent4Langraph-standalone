# 面试高频问题库 (Interview Q&A)

> 按岗位分类，覆盖 RAG/Agent 项目面试中 90% 的高频问题  
> 每个问题提供: 考察点 + 回答思路 + 参考话术

---

## 通用必问题（算法岗 & 开发岗）

### Q1: 介绍一下你这个项目？

**考察点**: 结构化表达能力、对项目的整体理解、能否抓住重点

**回答思路**（60 秒法则）:
1. 一句话定位（是什么，解决什么问题）
2. 技术栈（核心框架，2-3 个即可）
3. 你的核心贡献（不是团队做了什么，是你做了什么）
4. 量化成果（至少 1 个数字）

**参考话术**:
> "这是一个面向生产环境的企业级对话知识库系统。核心能力是每个对话拥有独立的文档集合，实现真正的会话隔离。技术上基于 LangGraph 状态机 + FastAPI + Vue3，我独立负责了检索层的设计和记忆管理模块。关键成果是混合检索使召回率达到 85%，P99 延迟控制在 2 秒以内，目前已服务 20+ 内部用户。"

---

### Q2: 项目的创新点 / 亮点在哪里？

**考察点**: 区分"调包"和"自研"的能力，技术判断力

**回答思路**:
- 算法岗: 指向算法创新（新策略、新结构、指标提升）
- 开发岗: 指向工程创新（架构设计、性能优化、可观测性）

**参考话术（算法岗）**:
> "传统 RAG 直接做向量检索，我们在三个点做了改进：一是引入查询重写，用 LLM 先做指代消解和子查询拆分，解决多跳问题；二是设计自适应检索深度，根据中间结果动态调整召回范围；三是改进 RRF 融合权重，除了语义相似度还考虑了文档时效性。这三个改进叠加使 F1 从 62% 提升到 85%。"

**参考话术（开发岗）**:
> "最大的工程亮点是状态一致性设计。用户可能中途中断生成，我们实现了脏 checkpoint 自动回滚，语义上'忽略本轮'，这个设计在 LangGraph 社区里参考案例很少。另一个是并发控制，我们用信号量限制全局 Ingestion 并发数，防止大文件上传打爆 LLM API 配额，上线后零配额溢出事故。"

---

### Q3: 如果数据量扩大 10 倍 / 用户量扩大 100 倍，怎么扩展？

**考察点**: 系统设计能力、分布式思维、对瓶颈的识别

**回答思路**（分层回答）:
1. 先定位当前架构的瓶颈（CPU/内存/IO/网络）
2. 存储层扩展（分片、读写分离、冷热分层）
3. 计算层扩展（水平扩容、负载均衡、缓存）
4. 链路优化（异步化、批处理、边缘缓存）

**参考话术**:
> "当前瓶颈可能在向量数据库和 LLM API 调用两方面。存储层：Chroma 单机版会变成瓶颈，可以迁移到 Milvus 分布式集群，按 conversation_id 做分片。API 层：FastAPI 可以水平扩容，配合 Nginx 负载均衡。LLM 调用：引入语义缓存（Redis）和请求合并（Batching），预计能降低 60-80% 的 API 调用量。Ingestion Pipeline：拆分为独立服务，用消息队列削峰，避免上传高峰期影响查询服务。"

---

### Q4: 出现过什么线上问题 / Bug？怎么解决的？

**考察点**: 工程经验、故障排查能力、复盘思维

**回答思路**（STAR）:
1. 现象: 用户反馈什么问题？
2. 根因: 技术层面根本原因（不要停留在表面）
3. 解决: 短期修复 + 长期预防
4. 复盘: 有什么机制避免再次发生？

**参考话术**:
> "上线初期遇到过一个奇怪的问题：用户上传 PDF 后偶尔检索不到内容。排查发现是 PDF 解析器对某些扫描版 PDF 提取文本为空，但前端显示'处理成功'。根因是异常处理不够细，解析失败没有阻断 Pipeline。短期修复：增加解析结果校验，空文本标记为 error 状态。长期：引入 OCR 兜底，扫描版自动走 OCR 分支。后续我们在 Pipeline 每个阶段都加了状态校验和熔断机制。"

---

### Q5: 怎么评估效果？指标是多少？

**考察点**: 数据思维、评估方法论、是否"凭感觉"

**回答思路**:
1. 检索侧指标（Recall@K、MRR、NDCG）
2. 生成侧指标（Faithfulness、Relevance、Hallucination Rate）
3. 业务侧指标（延迟、成本、用户满意度）
4. 人工评估（Golden Set、A/B Test）

**参考话术**:
> "我们分了三个层面评估。检索层：用 Recall@5 和 MRR，目前是 85% 和 0.72。生成层：用 Ragas 的 faithfulness 和 answer_relevancy，分别是 0.81 和 0.76。业务层：P99 延迟 1.8s，日均 API 成本约 50 元。另外我们建了 120 条的 Golden Test Set，覆盖简单问答、多跳推理、拒答三种场景，每版发布前跑回归测试。"

## 算法岗专项问题

### Q6: 为什么用 RRF 而不是 Linear 融合？做过对比实验吗？

**考察点**: 算法选型依据、实验设计能力

**参考话术**:
> "初期对比了三种融合策略：Linear、RRF 和 Learned（用 LightGBM 学权重）。Linear 对分数分布敏感，不同模型的相似度分数尺度不一致时效果不稳定。Learned 在小数据集上过拟合严重。RRF 不需要训练，对分数尺度不敏感，在我们的场景下 MRR 比 Linear 高 8%，所以最终选 RRF。消融实验显示去掉融合只做 Dense，MRR 下降 15%。"

---

### Q7: 你的 Reranker 是自己训的还是用的现成的？如果自己做，数据怎么来的？

**考察点**: 模型训练经验、数据工程能力

**参考话术（使用现成）**:
> "目前用的是现成的 BGE-Reranker，但做过领域内对比实验，在我们的测试集上比 Cross-Encoder 差 5%，计划下一步用 LoRA 微调。数据构造思路是：用现有系统的查询日志，取 Top-10 检索结果，人工标注相关/弱相关/不相关三级，预计需要 5000 条。"

**参考话术（自己训练）**:
> "自己用 LoRA 微调的。数据来源三部分：一是现有系统日志里的用户点击数据（弱监督）；二是人工标注的 2000 条困难样本对；三是对比学习生成的难负样本。训练用 InfoNCE 损失，batch size 32，LoRA rank 64，在 A100 上训了 3 个 epoch。最终 NDCG@10 从 0.58 提升到 0.76。"

---

### Q8: 有没有做过消融实验？贡献最大的模块是哪个？

**考察点**: 科学实验素养、对系统各模块的理解

**参考话术**:
> "做了完整的消融实验，控制单一变量。基线是纯 Dense 检索，F1=62%。加上 Sparse（BM25）后 F1=71%，提升 9%。加上 Reranker 后 F1=78%，提升 7%。加上查询重写后 F1=83%，提升 5%。加上自适应检索深度后 F1=85%，提升 2%。贡献最大的是混合检索部分， query 重写也很重要，但依赖查询复杂度，简单查询提升不明显。"

---

### Q9: 上下文窗口有限，怎么处理长文档？

**考察点**: 上下文工程、信息压缩能力

**参考话术**:
> "我们在三个层面处理。预处理层：用 Semantic Chunking 代替固定长度分块，保证语义完整性。检索层：RAG-Fusion，把长查询拆成子查询分别检索，再融合结果。生成层：如果检索结果总长度超窗口，先做层级摘要（chunk→section→document），优先保留高相关度的完整 chunk，低相关度的用摘要替代。这样在保证信息完整性的前提下，Token 使用降低 40%。"

---

### Q10:  hallucination 怎么解决？

**考察点**: 对 LLM 局限性的理解、缓解策略

**参考话术**:
> "我们用了四层防御。第一层：检索结果必须包含在上下文里，限制模型自由发挥（Grounding）。第二层：Prompt 里明确要求'只基于提供的文档回答，不确定就说不知道'。第三层：生成后用另一个模型做事实性校验（Claim Verification），对比生成内容和检索片段。第四层：用户可展开查看引用来源，人工快速校验。目前 Hallucination Rate 从初期的 15% 降到 4%。"

## 开发岗专项问题

### Q11: 为什么选择 LangGraph 而不是 LangChain / 其他框架？

**考察点**: 技术选型能力、对框架生态的理解

**参考话术**:
> "LangChain 适合简单链路，但我们的对话系统需要状态管理和循环控制（多轮交互、用户中断回滚），LangGraph 的图导向状态机更适合。也对比过 AutoGen，AutoGen 的 Multi-Agent 对话很强，但我们的场景是单 Agent 复杂工作流，LangGraph 的可控性更好。另外 LangGraph 的 Checkpoint 机制帮我们实现了记忆持久化，这是关键需求。"

---

### Q12: 文件上传后怎么处理？如果上传了一个 100MB 的 PDF 怎么办？

**考察点**: 系统工程、边界情况处理

**参考话术**:
> "文件上传是异步流程。前端上传后立即返回 file_id，后端放入任务队列，前端轮询状态。Pipeline 分 6 阶段：解析→分块→Refine→Enrich→Caption→向量化，每个阶段有独立状态。对于大文件：一是大小限制（单文件 50MB），超限拒绝；二是分页解析，避免一次性加载内存；三是全局并发信号量控制（同时只能处理 2 个文件），防止大文件占用过多资源；四是超时机制，单阶段超过 5 分钟标记失败。"

---

### Q13: 并发用户同时查询，怎么保证数据隔离？

**考察点**: 多租户设计、数据安全

**参考话术**:
> "核心设计是每个 conversation 对应独立的 Chroma collection，命名规则是 `conv_{conversation_id}`。查询时从请求头/参数中获取 conversation_id，严格限定检索范围。用户鉴权层验证该用户是否有权限访问这个 conversation。另外数据库层也有外键约束，文件、消息、记忆都关联到 conversation_id，物理上杜绝跨会话数据泄漏。"

---

### Q14: 怎么监控系统的健康状况？

**考察点**: 可观测性、运维意识

**参考话术**:
> "三层监控。日志层：结构化日志（JSON 格式），按 trace_id 串联全链路。指标层：Prometheus 采集 QPS、延迟、错误率、Token 消耗。展示层：Streamlit Dashboard 可视化，包括检索耗时分布、各阶段 Pipeline 成功率、每日 API 成本。告警：错误率 >5% 或 P99 延迟 >5s 时钉钉告警。另外每个对话可查看 LangGraph Trace，定位具体哪一步慢了。"

---

### Q15: 从 0 到 1 搭建这个系统，你的技术决策过程是什么？

**考察点**: 架构思维、决策依据、不是盲目堆砌技术

**参考话术**:
> "分三个阶段决策。MVP 阶段：目标是跑通核心链路，选 FastAPI+Chroma+OpenAI，快速验证需求。优化阶段：发现纯向量检索召回不够，引入 BM25 做混合检索；发现历史消息线性增长，引入滑动窗口压缩。生产阶段：要支持多用户，引入 PostgreSQL 做持久化；要对外暴露能力，引入 MCP 协议。每个技术选型都有明确的触发条件和量化验证，不是拍脑袋。"

## 压力测试问题

### Q16: 如果让你重做这个项目，有什么会不一样？

**考察点**: 反思能力、成长思维

**参考话术**:
> "有两个点会重新考虑。一是向量数据库选型，当时选 Chroma 是因为轻量，但现在看 Milvus 的分布式能力和混合检索原生支持更好，如果预期数据量 >100万 文档，应该直接上 Milvus。二是评估体系，初期太依赖主观判断，应该从一开始就建 Golden Test Set，这样每次改策略都有客观依据。这也说明评估和监控应该和核心功能同步建设，而不是事后补。"

### Q17: 你读过哪些相关论文？对你项目影响最大的是哪篇？

**考察点**: 技术深度、是否紧跟前沿

**参考话术（算法岗必问）**:
> "ReAct 论文（Yao et al., 2022）是我们 Agent 设计的理论基础，推理-行动交替的模式比单纯 CoT 更适合工具调用场景。Self-RAG（Asai et al., 2023）对我们的检索决策层启发很大，让模型自己判断是否需要检索、检索几次。最近 Anthropic 的《Building Effective Agents》也验证了我们在系统设计上的一些决策，比如'简单组合往往比复杂 Agent 更可靠'。"

### Q18: 你的项目和 xx 开源项目（如 Dify/LangFlow）比，优势在哪？

**考察点**: 竞品分析能力、差异化思维

**参考话术**:
> "Dify 是低代码平台，优势是快速搭建和丰富的预置组件，适合业务方自助。我们的定位是深度定制的企业级知识库，优势在三个点：一是对话级数据隔离，Dify 是知识库级隔离，我们做到 conversation 级；二是记忆管理深度，支持 checkpoint 回溯和长期记忆；三是 MCP 协议原生支持，可被外部 Agent 生态调用，不是封闭系统。简单说，Dify 是瑞士军刀，我们是手术刀。"
