# RAG / Agent 项目评判标准 (RAG-Agent Project Evaluation Rubric)

> **版本**: v1.0  
> **适用范围**: RAG 系统、Agent 应用、Multi-Agent 协作系统、对话式知识库等 LLM 应用项目  
> **目标**: 为求职者、面试官、开源贡献者提供一套可量化、可复现的项目评价框架

---

## 一、项目分级体系（四级模型）

| 级别 | 名称 | 定义 | 典型特征 | 简历价值 |
|:---|:---|:---|:---|:---|
| **L1** | 玩具级 (Toy) | 跑通教程的 Demo，无实际场景 | 单文件脚本、硬编码参数、无错误处理、无测试 | ❌ 不建议写入 |
| **L2** | 课程级 (Course) | 完成课程作业或练手项目 | 基础功能完整、有简单封装、缺少生产化考虑 | ⚠️ 可写，但需包装 |
| **L3** | 生产级 (Production) | 可部署运行的企业应用 | 完整链路、异步处理、错误恢复、可观测性、测试覆盖 | ✅ 强烈推荐写入 |
| **L4** | 研究级 (Research) | 具备算法创新或技术突破 | 论文级创新、消融实验、SOTA 指标、开源影响力 | ✅✅ 核心竞争力 |

> **求职底线**: 简历中的项目不得低于 **L2+**；目标大厂应至少有一个 **L3** 项目。

---

## 二、评判维度与权重

总分 100 分，按以下五个维度评分：

```
┌─────────────────────────────────────────────────────────┐
│  工程架构与代码质量 ........... 30% (30分)               │
│  算法深度与技术创新 ........... 25% (25分)               │
│  产品完整度与用户体验 ......... 20% (20分)               │
│  可观测性、评估与实验 ......... 15% (15分)               │
│  文档质量与开源影响力 ......... 10% (10分)               │
└─────────────────────────────────────────────────────────┘
```

---

## 三、维度一：工程架构与代码质量（30分）

### 3.1 架构设计（10分）

| 评分 | 标准 | 检查点 |
|:---|:---|:---|
| 1-2分 | 无架构可言 | 单文件/单模块，所有逻辑耦合在一起 |
| 3-4分 | 基础分层 | 至少有 `model/service/controller` 分层，但边界模糊 |
| 5-6分 | 清晰模块化 | 各模块职责单一，接口定义清晰，依赖关系合理 |
| 7-8分 | 领域驱动/微服务 | 按业务域拆分（如 Ingestion / Retrieval / Generation 独立） |
| 9-10分 | 企业级架构 | 支持水平扩展、服务降级、熔断、多租户、插件化扩展 |

**关键检查项**:
- [ ] 是否采用工厂模式管理多 Provider（LLM / Embedding / Reranker）?
- [ ] 是否有清晰的 Pipeline / Workflow 编排（如 LangGraph、Airflow）?
- [ ] 存储层是否分离（向量库、关系库、缓存各司其职）?
- [ ] 是否支持配置化，无需改代码即可切换组件?

### 3.2 异步与并发（6分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 全同步阻塞 |
| 2分 | 简单异步（如 `async def` 但未做并发控制） |
| 4分 | 有并发控制（信号量、连接池、限流） |
| 6分 | 完整的异步链路 + 背压机制 + 队列削峰 |

**关键检查项**:
- [ ] 文件上传是否后台异步处理，前端轮询状态?
- [ ] 是否有并发控制防止 API 配额被打爆?
- [ ] 耗时操作是否使用任务队列（Celery / RQ / 自研）?

### 3.3 错误处理与容错（6分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无 try-catch，出错直接 500 |
| 2分 | 基础异常捕获，统一错误返回 |
| 4分 | 分级容错（网络超时重试、模型降级、默认兜底） |
| 6分 | 事务级一致性（如中断后状态回滚、脏数据清理） |

**关键检查项**:
- [ ] LLM 调用失败是否有降级策略（切换模型 / 返回提示）?
- [ ] 向量化失败是否标记状态，支持断点续传?
- [ ] 用户中途取消生成，是否能优雅终止并清理资源?

### 3.4 代码质量（4分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无类型注解、无文档字符串、命名混乱 |
| 1分 | 基础规范（PEP8、类型注解覆盖 > 50%） |
| 2分 | 良好规范 + Docstring + 静态检查（mypy/ruff） |
| 3分 | 有代码审查配置（pre-commit、CI  lint） |
| 4分 | 架构决策记录（ADR）、接口契约测试 |

### 3.5 测试覆盖（4分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无任何测试 |
| 1分 | 仅有手工测试记录 |
| 2分 | 单元测试覆盖核心模块（> 60%） |
| 3分 | 三层测试（Unit + Integration + E2E） |
| 4分 | 契约测试 + 性能基准测试 + 混沌测试 |

---

## 四、维度二：算法深度与技术创新（25分）

### 4.1 RAG 策略复杂度（10分）

| 评分 | 检索策略 | 代表能力 |
|:---|:---|:---|
| 1-2分 | 基础 RAG | 直接向量检索 + 简单 Prompt |
| 3-4分 | 高级 RAG | 混合检索（Dense+Sparse）、Rerank、查询重写 |
| 5-6分 | Modular RAG | 路由决策（Route）、主动改写（HyDE）、多路召回 |
| 7-8分 | Agentic RAG | Agent 自主决策检索策略、多跳推理、Self-RAG |
| 9-10分 | GraphRAG / 多模态 RAG | 知识图谱增强、跨模态检索、自研索引算法 |

**关键检查项**:
- [ ] 是否实现查询重写（Query Rewriting / Expansion）?
- [ ] 是否有检索结果的后处理（去重、重排序、截断策略）?
- [ ] 是否实现多跳推理（Multi-hop Retrieval）?
- [ ] 是否支持上下文压缩（Context Compression / RAG-Fusion）?

### 4.2 自研算法/策略（8分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 纯调包，无任何自研逻辑 |
| 2分 | 有自研 Prompt 模板或简单的后处理规则 |
| 4分 | 自研分块策略、自定义召回逻辑、改进的 RRF 权重 |
| 6分 | 自研轻量级模型（如用 LoRA 微调 Reranker / Embedding） |
| 8分 | 发表论文级创新（新架构、新损失函数、新训练范式） |

### 4.3 记忆与上下文管理（4分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无记忆，或简单拼接历史 |
| 1分 | 固定窗口历史（如最近 10 轮） |
| 2分 | 滑动窗口 + 摘要压缩 |
| 3分 | 多级记忆（STM / MTM / LTM） |
| 4分 | 记忆个性化（用户画像、长期偏好学习） |

### 4.4 模型训练与微调（3分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 仅调用第三方 API |
| 1分 | 使用 LoRA/QLoRA 微调过模型 |
| 2分 | 完整训练流程（数据构造 → 训练 → 评估 → 部署） |
| 3分 | 强化学习优化（PPO / DPO / GRPO 对齐 Agent 行为） |

## 五、维度三：产品完整度与用户体验（20分）

### 5.1 功能完整性（8分）

| 评分 | 功能覆盖 |
|:---|:---|
| 1-2分 | 仅支持单轮问答 |
| 3-4分 | 多轮对话 + 文件上传 |
| 5-6分 | 对话管理（CRUD）、文件管理、历史记录、来源展示 |
| 7-8分 | 用户系统、权限管理、知识库分享、API 开放平台 |

**关键检查项**:
- [ ] 是否支持多格式文件上传（PDF/DOCX/CSV/MD）?
- [ ] 是否有对话级别的数据隔离（多租户）?
- [ ] AI 回复是否展示引用来源（Citation）?
- [ ] 是否支持流式输出（SSE / WebSocket）?

### 5.2 前端与交互（6分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无前端，仅 CLI / API |
| 2分 | 基础前端（如 Gradio / Streamlit） |
| 4分 | 现代化前端（Vue/React），响应式设计 |
| 6分 | 专业级交互（拖拽上传、Markdown 渲染、代码高亮、LaTeX、思维导图） |

### 5.3 部署与运维（6分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 仅本地运行 |
| 2分 | Docker 化，可一键启动 |
| 4分 | 有 Docker Compose / K8s 编排，CI/CD 流水线 |
| 6分 | 云原生部署（自动扩缩容、监控告警、日志聚合、成本优化） |

## 六、维度四：可观测性、评估与实验（15分）

### 6.1 评估体系（8分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无评估，凭感觉说"效果好" |
| 2分 | 人工抽查，有简单的 Good/Bad 标记 |
| 4分 | 使用自动化评估框架（Ragas / ARES / TruLens） |
| 6分 | 自建 Golden Test Set，覆盖多场景（简单/多跳/拒答） |
| 8分 | 多维度评估 + LLM-as-Judge + 人工校验 + 持续回归 |

**关键检查项**:
- [ ] 是否有固定的测试集（> 50 条）?
- [ ] 是否评估检索侧指标（Recall@K、MRR、NDCG）?
- [ ] 是否评估生成侧指标（Faithfulness、Relevance、Hallucination Rate）?
- [ ] 是否有成本评估（Token 消耗、API 费用、端到端延迟）?

### 6.2 实验与对比（4分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 无实验 |
| 1分 | 简单的前后对比（"优化后更快了"） |
| 2分 | 消融实验（Ablation Study），控制单一变量 |
| 3分 | 多 Baseline 对比（与原始 RAG / 竞品系统对比） |
| 4分 | 完整的实验报告（假设 → 方法 → 数据 → 结论 → 局限） |

### 6.3 可观测性（3分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 黑盒运行 |
| 1分 | 打印日志 |
| 2分 | 结构化日志 + Trace（如 LangSmith / LangFuse） |
| 3分 | 可视化 Dashboard + 链路追踪 + 成本监控 + 告警 |

### 7.1 技术文档（4分）

|:---|:---|

|:---|:---|

### 7.3 开源影响力（3分）

| 评分 | 标准 |
|:---|:---|
| 0分 | 私有仓库或未开源 |

| 2分 | 有社区讨论（Issue/PR），10-100 Stars |
| 3分 | 被他人引用/复用，> 100 Stars，或官方推荐 |

## 八、岗位适配评分矩阵

不同岗位的评分侧重点不同，换算最终得分时加权如下：

| 维度 | 算法岗权重 | 开发岗权重 |
|:---|:---:|:---:|
| 工程架构与代码质量 | 20% | **40%** |
| 算法深度与技术创新 | **40%** | 15% |
| 产品完整度与用户体验 | 15% | 25% |
| 可观测性、评估与实验 | **20%** | 15% |
| 文档质量与开源影响力 | 5% | 5% |

### 算法岗适配评分标准

总分 = 工程 × 0.2 + 算法 × 0.4 + 产品 × 0.15 + 评估 × 0.2 + 文档 × 0.05

| 总分区间 | 等级 | 求职建议 |
|:---|:---|:---|
| 85-100 | S 级 | 可冲刺大厂核心算法岗 |
| 70-84 | A 级 | 适合投递主流算法岗 |
| 55-69 | B 级 | 可投，但需补充算法创新点 |
| 40-54 | C 级 | 建议作为辅助项目，另备核心算法项目 |
| < 40 | D 级 | 不建议作为算法岗核心项目 |

**算法岗加分项（额外 +5~10 分）**:
- 有论文在投或已发表（顶会/期刊）
- 开源代码被引用或复现
- 在公开 Benchmark 上有排名
- 实现了新的损失函数或训练策略

### 开发岗适配评分标准

总分 = 工程 × 0.4 + 算法 × 0.15 + 产品 × 0.25 + 评估 × 0.15 + 文档 × 0.05

| 总分区间 | 等级 | 求职建议 |
|:---|:---|:---|
| 85-100 | S 级 | 可冲刺大厂高级开发 / 架构师 |
| 70-84 | A 级 | 适合投递主流 RAG/Agent 开发岗 |
| 55-69 | B 级 | 可投，但需补充业务指标和部署经验 |
| 40-54 | C 级 | 建议完善测试和文档后再投递 |
| < 40 | D 级 | 需要重大重构 |

**开发岗加分项（额外 +5~10 分）**:
- 有真实用户或业务数据（如服务 1000+ 用户）
- 性能优化数据（QPS 提升 X 倍、延迟降低 X%）
- 云原生部署经验（K8s / Serverless）
- 生产故障处理案例

## 九、快速检查清单 (Quick Checklist)

### 9.1 工程底线（不满足 = L1/L2）

- [ ] 项目可以在新机器上按 README 步骤 10 分钟内跑通
- [ ] 有 requirements.txt / pyproject.toml / package.json 等依赖声明
- [ ] 核心模块有单元测试，且测试可以通过
- [ ] 没有硬编码的 API Key、密码、路径
- [ ] 有 .gitignore，不提交临时文件和敏感数据

### 9.2 生产级门槛（满足 = L3）

- [ ] 有异步处理机制，文件上传不阻塞主流程
- [ ] 有错误处理和降级策略，不会因单点故障整体崩溃
- [ ] 有配置化管理，不同环境（dev/test/prod）可切换
- [ ] 有可观测性（日志 / Trace / Dashboard 至少其一）
- [ ] 有自动化评估或固定测试集
- [ ] 前后端分离，或至少有清晰的前端界面

### 9.3 研究级门槛（满足 = L4）

- [ ] 有自研算法或策略，不仅仅是调用第三方库
- [ ] 有消融实验或多 Baseline 对比
- [ ] 有公开的量化指标，且优于原始基线
- [ ] 有详细的技术文档说明创新点
- [ ] 代码开源，有社区互动（Issue/PR/Discussion）

## 十一、使用建议

### 对求职者
1. **自评**: 用本标准给手头项目打分，低于 55 分的项目不建议写入简历
2. **定位**: 根据目标岗位选择权重矩阵，针对性补强弱项
3. **包装**: 将已有能力映射到评分维度，用标准语言描述项目（如"实现了模块化 RAG Pipeline，支持热插拔 Reranker"）

### 对面试官
1. **初筛**: 用"快速检查清单"判断项目级别，低于 L2 的可以直接过滤
2. **深挖**: 按维度追问，如"你的错误处理策略是什么？""有没有做消融实验？"
3. **校准**: 警惕"高分低能"（文档华丽但代码粗糙）和"低分高能"（研究型代码工程化不足）

### 对开源维护者
1. **路线图**: 用评分找到短板，制定明确的升级路径
2. **展示**: 在 README 中主动展示量化指标（评估分数、性能数据、测试覆盖率）
3. **社区**: 补充求职导向文档，降低用户理解成本
